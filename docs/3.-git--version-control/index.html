    3. Git & Version Control#

    10. Git — Core Benefit#

    • Tracks changes to files over time
    • Enables collaboration among multiple team members
    • Allows rollback to any previous state
    • ❌ Does NOT automatically clean data
    • ❌ Does NOT replace testing
    • ❌ Does NOT store only binary assets

    11. .gitignore File#

    • Specifies files/folders that Git should NOT track
    • Common entries:
    __pycache__/
.env          # API keys, secrets
*.pyc
node_modules/
data/raw/     # large data files
    • ❌ NOT for tracking changes to specific files
    • ❌ NOT for defining repository settings (that’s .git/config)
    • ❌ NOT for listing contributors

    12. git diff — View Changes#

    • Shows differences between working directory and last commit
    • Useful before staging to review what changed
    • Related commands:
      • git status → shows which files are modified/staged (NOT the actual diff)
      • git log → shows commit history
      • git show → shows changes in a specific commit

    13. Correct Git Sequence — Save & Upload#

    git add .  →  git commit -m "message"  →  git push
    • git add . → stages all changed files
    • git commit -m → saves snapshot locally
    • git push → uploads to remote (GitHub)
    • ❌ git push → git add → git commit → wrong order
    • ❌ git merge → git push → skips staging and committing

    14. git pull origin main — Sync with Remote#

    • Downloads and integrates latest updates from remote main branch
    • Use before pushing your own changes to avoid conflicts
    • Run while on your feature branch:
    git pull origin main
    • ❌ git commit main → not a valid command
    • ❌ git init main → initializes a new repo, doesn’t sync

    15. Git Branching — Create & Switch#

    # Two-step approach
git branch feature-analysis
git checkout feature-analysis

# One-step shortcut
git checkout -b feature-analysis
    • Always create feature branches — never work directly on main
    • ❌ git commit feature-analysis → not valid
    • ❌ git push feature-analysis → pushes but doesn’t create locally

    16. git checkout -- . — Discard Changes#

    • Discards all uncommitted changes in working directory
    • Reverts files back to last committed state
    git checkout -- .        # discard all changes
git checkout -- file.py  # discard changes in specific file
    • ❌ git revert HEAD → creates a new commit that undoes last commit (different!)
    • ❌ git reset --hard origin/master → resets to remote state (more destructive)
    • ❌ git clean -fd → removes untracked files only

    17. Pull Requests (PR)#

    • Formal request to merge your branch into main
    • Allows team members to:
      • Review your code
      • Suggest changes
      • Approve before merging
    • Best practice in collaborative data science projects
    • ❌ PRs are NOT faster than direct merging
    • ❌ PRs do NOT automatically fix bugs

    18. Merge Conflicts#

    • Occur when two people edit the same line on different branches
    • Git cannot auto-resolve → must be resolved manually
    • Resolution process:
      • Open conflicted file
      • Choose which change to keep (or combine both)
      • Stage and commit the resolved file
    • ❌ Git does NOT automatically keep newer change
    • ❌ Git does NOT delete both changes

    19. Git Hotfix Workflow#

    • For critical production bugs that need immediate fix:
    1. Create hotfix branch from last release tag
2. Implement and test the fix
3. Open PR for expedited review
4. Merge into both main AND develop
    • ❌ Do NOT apply fix directly to main without review
    • ❌ Do NOT delay fix to next release cycle if critical

    20. Full Collaborative Git Workflow#

    git add .
→ git commit -m "message"
→ git push
→ Create Pull Request
→ Code Review
→ Merge into main
    • Feature branches keep main clean and stable
    • main = production-ready code
    • dev/develop = development/testing code
    • Feature branches = individual work

    21. Pipeline Transformation Versioning#

    • When transformation logic changes frequently:
      • ✅ Version transformations with git
      • ✅ Write unit tests for each transformation
      • ✅ Test on sample data before pushing to production
    • ❌ Don’t keep all logic in one script and update directly
    • ❌ Don’t manually document in shared docs only
    • ❌ Don’t avoid changes to prevent breaking things
